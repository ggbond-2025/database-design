# openGauss 数据库课程设计验收准备

本文基于项目实际代码和 SQL 脚本整理，主要对应 `design/sql/02_schema.sql`、`03_views.sql`、`04_triggers.sql`、`05_procedures.sql` 和 `角色功能说明.md`。文档按验收时可能被问到的问题分类放置，便于答辩时快速定位回答。

## 数据库设计

**问题：本项目数据库的整体定位是什么？**

本项目使用 openGauss，数据库名为 `DengjxMIS13`。表名、字段名统一带有 `Dengjx`、`djx` 和组号 `13`，用于满足课程设计个性化命名要求。

数据库围绕高校教务系统设计，主要覆盖基础资料、教务业务和系统支撑三类数据：

1. 基础资料：地区、专业、班级、学生、教师、课程、教学楼、教室。
2. 教务业务：专业课程计划、开课安排、选课、成绩、期末考试、教学评价、转专业。
3. 系统支撑：选课开关、转专业开关、用户账号、统计视图、触发器、存储过程。

**问题：为什么不把课程、教师、班级、上课时间都放在课程表里？**

课程表只保存课程基础信息，例如课程编码、课程名称、学时、考核方式和学分。某学期某班由某教师在某教室上课，是一次具体开课行为，由开课安排表描述。

这样拆分的原因是同一门课程可能在不同专业、不同班级、不同学期重复开设。如果把教师、班级、教室、上课时间直接放进课程表，会产生大量冗余，也不利于维护。当前设计把可复用的课程定义和每学期发生的教学任务分开，更符合教务系统实际业务。

**问题：数据库设计的核心层次是什么？**

核心层次可以概括为：

1. `Dengjx_Courses13` 表示课程基础定义。
2. `Dengjx_MajorCourses13` 表示某专业的培养方案。
3. `Dengjx_TeachingAssignments13` 表示某学期的具体开课。
4. `Dengjx_Enrollments13` 表示学生是否参与某次开课。
5. `Dengjx_Grades13` 和 `Dengjx_TeachingEvaluations13` 分别表示成绩和评价。

整体上，基础定义和业务发生分离，选课、成绩、评价都围绕具体开课安排展开。

**问题：主要数据表有哪些，分别解决什么问题？**

| 表名 | 作用 | 建表原因 |
|---|---|---|
| `Dengjx_Regions13` | 地区信息表，保存学生生源地。 | 地区独立维护，学生只引用地区编号，便于统一维护地区名称，也便于按地区统计学生数。 |
| `Dengjx_Majors13` | 专业信息表，保存专业名称和毕业学分要求。 | 专业是班级、培养方案、转专业和学分统计的基础。毕业学分放在专业表中，便于和学生已修学分对比。 |

| `Dengjx_EnrollmentSettings13` | 选课设置表，保存全局选课开关。 | 控制学生端是否允许选课和退课，固定主键为 1，表示系统级单例配置。 |
| `Dengjx_MajorTransferSettings13` | 转专业申请设置表，保存全局转专业申请开关。 | 控制学生端是否允许提交转专业申请，同样是系统级单例配置。 |
| `Dengjx_TeachingAssignments13` | 教学任务/开课安排表，关联专业课程计划、班级、教师、教室、学年学期、容量、选课开放状态和上课时间。 | 这是教务业务核心表。课程只是基础定义，开课安排才表示某学期某班由某教师在某教室上的具体课程。 |
| `Dengjx_MajorTransferApplications13` | 转专业申请表，保存申请学生、原专业/班级、目标专业/班级、审核状态、审核意见和生效学期。 | 用于完整保留学生转专业申请和审核过程。审核通过后只更新学生当前班级，不删除历史选课、成绩和已完成课程。 |
| `Dengjx_Enrollments13` | 选课记录表，关联学生和开课安排，保存选课状态和选课/退选时间。 | 表示学生是否参与某个开课安排。成绩和教学评价都以选课记录为基础，避免未选课学生出现成绩。 |
| `Dengjx_TeachingEvaluations13` | 教学评价表，关联选课记录，保存评分、评价理由和提交时间。 | 每条选课记录最多评价一次，保证学生对已完成课程只提交一次最终评价。 |
| `Dengjx_Grades13` | 成绩表，关联选课记录，保存课程成绩和录入时间。 | 成绩必须基于有效选课记录，一条选课记录最多一条成绩。 |
| `Dengjx_Users13` | 用户账号表，保存登录用户名、密码哈希、角色、绑定学生/教师和账号状态。 | 账号与人员档案分离。管理员不绑定档案，学生账号绑定学生，教师账号绑定教师。 |

**问题：主要表之间是什么关系？**

一个专业可以包含多个班级，一个班级可以包含多个学生，学生同时关联班级和生源地区。专业和课程是多对多关系，通过专业课程计划表关联。开课安排由专业课程计划、班级、教师、教室、学年和学期共同确定。学生通过选课记录参与开课安排，选课记录最多对应一条成绩和一条教学评价。用户账号和人员档案分离，学生账号绑定学生档案，教师账号绑定教师档案，管理员账号不绑定人员档案。

## 学生选课信息表联动

**问题：一个学生的选课信息是如何实现的，哪几张表联动？**

一个学生的选课信息不是单表保存完整课程详情，而是以 `Dengjx_Enrollments13` 选课记录表为核心，再联动学生、开课安排、课程计划、课程、教师和教室等表查询出来。

核心表关系可以概括为：

```text
Dengjx_Users13
  -> Dengjx_Students13
  -> Dengjx_Enrollments13
  -> Dengjx_TeachingAssignments13
  -> Dengjx_MajorCourses13
  -> Dengjx_Courses13
  -> Dengjx_Teachers13
  -> Dengjx_Classrooms13
  -> Dengjx_TeachingBuildings13
```

首先，学生登录账号保存在 `Dengjx_Users13` 中。学生账号的 `djx_StudentId13` 会绑定到 `Dengjx_Students13.djx_StudentId13`，因此后端可以根据当前登录用户找到对应学生档案。

真正表示“某个学生选择了某次开课”的表是 `Dengjx_Enrollments13`。该表通过 `djx_StudentId13` 关联学生，通过 `djx_AssignmentId13` 关联开课安排，并保存选课状态、选课时间和退课时间。其中 `djx_Status13` 只能是 `SELECTED`、`DROPPED`、`COMPLETED`，分别表示已选、已退选和已完成。表中还设置了 `UNIQUE (djx_StudentId13, djx_AssignmentId13)`，防止同一个学生重复选择同一个开课安排。

选课表只保存学生和开课安排之间的关系，不重复保存课程名、教师名、教室名等信息。查询“我的选课”时，系统从 `Dengjx_Enrollments13` 出发联查多张表：

1. 联查 `Dengjx_TeachingAssignments13`，获得学年、学期、容量、开放状态、上课时间、教师和教室等具体开课信息。
2. 联查 `Dengjx_MajorCourses13`，获得课程属于必修还是选修，以及建议修读年级和学期。
3. 联查 `Dengjx_Courses13`，获得课程编号、课程名称、学时、考核方式和学分。
4. 联查 `Dengjx_Teachers13`，获得任课教师信息。
5. 联查 `Dengjx_Classrooms13` 和 `Dengjx_TeachingBuildings13`，获得上课地点。

后端 `EnrollmentServiceImpl.mine()` 查询学生本人选课时，只返回 `SELECTED` 和 `COMPLETED` 状态的记录，退选状态 `DROPPED` 不进入“我的课程”和课表。这样可以保证学生退课后，课程不会继续显示在有效课程清单和课表中。

必修课和选修课的进入方式不同。必修课由 `autoEnrollRequiredCourses()` 根据学生当前班级、当前学年学期、当前年级学期自动补齐选课记录；选修课由学生在可选课程页面手动选择后写入 `Dengjx_Enrollments13`。因此，最终无论是必修课还是选修课，都会落到同一张选课记录表中统一管理。

成绩也通过选课记录联动。`Dengjx_Grades13` 使用 `djx_EnrollmentId13` 关联某条选课记录，表示成绩必须基于一次有效选课产生。教师或管理员录入成绩后，后端会把对应选课记录状态更新为 `COMPLETED`。学生成绩视图 `V_Dengjx_StudentGrades13` 也是从成绩表联查选课表、学生表、开课安排表、课程表和教师表得到完整成绩明细。

所以，一个学生的选课信息以 `Dengjx_Enrollments13` 为中心：左边连接登录账号和学生档案，右边连接开课安排、课程计划、课程、教师和地点；状态字段再控制这条记录是已选、退选还是已完成。

## 课程时间冲突判断

课程时间冲突不是通过数据库唯一约束直接完成，而是在后端业务层判断。原因是时间段重叠属于范围比较，普通唯一约束只能判断字段值是否完全相同，不能表达“两个时间区间是否相交”。数据库层主要负责基础完整性，例如上课时间字段必须完整填写，星期必须在周一到周五范围内，开始时间必须早于结束时间；真正的教师、班级、教室和学生选课时间冲突由 Java 代码判断。

后端把冲突判断的公共规则集中在 `AssignmentScheduleRules` 中。该类中定义了一个内部 `record`：

```java
public record EffectiveSlot(
        int weekday,
        LocalTime startTime,
        LocalTime endTime,
        boolean firstHalf,
        boolean secondHalf) {
}
```

`EffectiveSlot` 不是数据库表，也不是持久化实体，而是后端用于冲突判断的临时数据结构。它表示“一段真正参与判断的有效上课时间”，字段含义如下：

1. `weekday`：星期几，项目中 `1` 到 `5` 表示周一到周五。
2. `startTime`：这段课的开始时间。
3. `endTime`：这段课的结束时间。
4. `firstHalf`：这段课是否在前半学期有效。
5. `secondHalf`：这段课是否在后半学期有效。

项目不是直接拿数据库中的一行开课安排判断冲突，而是先根据课程学时，把开课安排转换成一个或多个 `EffectiveSlot`。转换代码在 `effectiveSlots()` 中：

```java
public static List<EffectiveSlot> effectiveSlots(Assignment assignment, int hours) {
    List<EffectiveSlot> slots = new ArrayList<>();
    addSlot(slots, assignment.getWeekdayOne(), assignment.getStartTimeOne(), assignment.getEndTimeOne(), true, hours != 32);
    if (hours >= 48) {
        addSlot(slots, assignment.getWeekdayTwo(), assignment.getStartTimeTwo(), assignment.getEndTimeTwo(), true, hours == 64);
    }
    return slots;
}
```

这里的学时规则是：

1. `32` 学时课程：只使用第一个上课时间，并且只在前半学期有效。因为 `hours != 32` 为 `false`，所以第一个时间段的 `secondHalf` 为 `false`。
2. `48` 学时课程：使用两个上课时间。第一个时间段全学期有效；第二个时间段只在前半学期有效。因为 `hours == 64` 为 `false`，所以第二个时间段的 `secondHalf` 为 `false`。
3. `64` 学时课程：使用两个上课时间，并且两个时间段都全学期有效。因为 `hours != 32` 和 `hours == 64` 都为 `true`，所以两个时间段都覆盖前半学期和后半学期。

`addSlot()` 会过滤掉不完整时间段：

```java
private static void addSlot(
        List<EffectiveSlot> slots,
        Integer weekday,
        LocalTime startTime,
        LocalTime endTime,
        boolean firstHalf,
        boolean secondHalf) {
    if (weekday == null || startTime == null || endTime == null) {
        return;
    }
    slots.add(new EffectiveSlot(weekday, startTime, endTime, firstHalf, secondHalf));
}
```

也就是说，如果星期、开始时间、结束时间任意一个为空，这个时间段不会进入冲突判断。正常创建或修改开课安排前，`AssignmentServiceImpl.validateSlot()` 会先校验时间段必须完整，所以业务入口处一般不会产生半截时间段。

真正判断两个时间段是否冲突的是 `EffectiveSlot.overlaps()`：

```java
private boolean overlaps(EffectiveSlot other) {
    return weekday == other.weekday
            && ((firstHalf && other.firstHalf) || (secondHalf && other.secondHalf))
            && startTime.isBefore(other.endTime)
            && other.startTime.isBefore(endTime);
}
```

这个判断要求同时满足三个条件：

1. 星期相同。
2. 有共同有效学期段，也就是同在前半学期有效，或者同在后半学期有效。
3. 时间区间发生交叉，即当前开始时间早于对方结束时间，并且对方开始时间早于当前结束时间。

因此，如果一门课在 `09:40` 结束，另一门课在 `09:40` 开始，不算时间冲突。因为第一门课的结束时间等于第二门课的开始时间，不满足 `other.startTime.isBefore(endTime)`。

多个时间段之间的冲突由 `hasConflict()` 完成：

```java
public static boolean hasConflict(List<EffectiveSlot> left, List<EffectiveSlot> right) {
    for (EffectiveSlot leftSlot : left) {
        for (EffectiveSlot rightSlot : right) {
            if (leftSlot.overlaps(rightSlot)) {
                return true;
            }
        }
    }
    return false;
}
```

这里采用双重循环，把当前课程的每一个有效时间段与另一门课程的每一个有效时间段逐个比较，只要任意一组 `EffectiveSlot` 重叠，就认为两门课存在冲突。

管理员创建或修改开课安排时，入口在 `AssignmentServiceImpl.create()` 和 `AssignmentServiceImpl.update()`。两个方法都会先调用 `validate()`：

```java
public Assignment create(AssignmentRequest request) {
    validate(request, null);
    Assignment assignment = new Assignment();
    apply(request, assignment);
    assignmentMapper.insert(assignment);
    return assignment;
}

public Assignment update(Long id, AssignmentRequest request) {
    validate(request, id);
    Assignment assignment = getById(id);
    apply(request, assignment);
    assignmentMapper.updateById(assignment);
    return assignment;
}
```

`validate()` 先检查学期、容量、上课时间完整性和课程学时要求，然后分别检查三类资源冲突：

```java
validateScheduleConflict(request, excludeAssignmentId, hours, ScheduleResource.TEACHER);
validateScheduleConflict(request, excludeAssignmentId, hours, ScheduleResource.CLASS);
validateScheduleConflict(request, excludeAssignmentId, hours, ScheduleResource.CLASSROOM);
```

三类资源分别表示：

1. `TEACHER`：同一教师同一学期不能在重叠时间上两门课。
2. `CLASS`：同一班级同一学期不能在重叠时间上两门课。
3. `CLASSROOM`：同一教室同一学期不能在重叠时间上两门课。

具体检查逻辑在 `validateScheduleConflict()`：

```java
Assignment candidate = new Assignment();
apply(request, candidate);
List<AssignmentScheduleRules.EffectiveSlot> candidateSlots =
        AssignmentScheduleRules.effectiveSlots(candidate, hours);
if (candidateSlots.isEmpty()) {
    return;
}

LambdaQueryWrapper<Assignment> wrapper = new LambdaQueryWrapper<Assignment>()
        .eq(Assignment::getAcademicYear, request.academicYear())
        .eq(Assignment::getSemester, request.semester());
resource.apply(wrapper, request);
if (excludeAssignmentId != null) {
    wrapper.ne(Assignment::getAssignmentId, excludeAssignmentId);
}

List<Assignment> existingAssignments = assignmentMapper.selectList(wrapper);
for (Assignment existing : existingAssignments) {
    if (!resource.matches(existing, request)) {
        continue;
    }
    Integer existingHours = courseHours(existing.getMajorCourseId());
    if (existingHours == null) {
        continue;
    }
    if (AssignmentScheduleRules.hasConflict(
            candidateSlots,
            AssignmentScheduleRules.effectiveSlots(existing, existingHours))) {
        throw new BusinessException(resource.conflictMessage);
    }
}
```

这段代码的实际流程是：

1. 先把本次要新增或修改的开课安排封装成 `candidate`。
2. 根据本课程学时，把 `candidate` 转成 `candidateSlots`。
3. 查询同一学年、同一学期下同一资源的已有开课安排。
4. 如果是修改操作，通过 `wrapper.ne(Assignment::getAssignmentId, excludeAssignmentId)` 排除当前正在修改的开课安排，避免和自己比较。
5. 遍历已有开课安排，查询已有课程的学时。
6. 把已有开课安排也转换成 `EffectiveSlot` 列表。
7. 调用 `AssignmentScheduleRules.hasConflict()` 做时间重叠判断。
8. 如果冲突，抛出对应业务异常，例如教师任课时间冲突、班级上课时间冲突或教室使用时间冲突。

`ScheduleResource` 枚举负责把“检查哪类资源”抽象出来：

```java
private enum ScheduleResource {
    TEACHER("教师任课时间冲突：同一学期内该教师已有重叠课程") {
        void apply(LambdaQueryWrapper<Assignment> wrapper, AssignmentRequest request) {
            wrapper.eq(Assignment::getTeacherId, request.teacherId());
        }
        boolean matches(Assignment assignment, AssignmentRequest request) {
            return request.teacherId().equals(assignment.getTeacherId());
        }
    },
    CLASS("班级上课时间冲突：同一学期内该班级已有重叠课程") {
        void apply(LambdaQueryWrapper<Assignment> wrapper, AssignmentRequest request) {
            wrapper.eq(Assignment::getClassId, request.classId());
        }
        boolean matches(Assignment assignment, AssignmentRequest request) {
            return request.classId().equals(assignment.getClassId());
        }
    },
    CLASSROOM("教室使用时间冲突：同一学期内该教室已有重叠课程") {
        void apply(LambdaQueryWrapper<Assignment> wrapper, AssignmentRequest request) {
            wrapper.eq(Assignment::getClassroomId, request.classroomId());
        }
        boolean matches(Assignment assignment, AssignmentRequest request) {
            return request.classroomId().equals(assignment.getClassroomId());
        }
    };
}
```

这样写的好处是冲突算法只写一份，教师、班级、教室只是查询条件和错误提示不同，不需要复制三套判断代码。

学生选课时也复用了同一套时间冲突算法。学生点击选课后，入口在 `EnrollmentServiceImpl.enroll()`。该方法会依次检查选课开关、课程是否开放、是否为选修课、是否属于当前年级学期、容量是否已满，最后调用：

```java
ensureNoScheduleConflict(context, assignment, numberValue(course.get("djx_hours13")).intValue());
```

`ensureNoScheduleConflict()` 的实现如下：

```java
private void ensureNoScheduleConflict(StudentTermContext context, Assignment target, int targetHours) {
    List<AssignmentScheduleRules.EffectiveSlot> targetSlots = AssignmentScheduleRules.effectiveSlots(target, targetHours);
    if (targetSlots.isEmpty()) {
        return;
    }
    List<Map<String, Object>> rows = jdbcTemplate.queryForList("""
            SELECT a.djx_assignmentid13,
                   a.djx_weekdayone13,
                   a.djx_starttimeone13,
                   a.djx_endtimeone13,
                   a.djx_weekdaytwo13,
                   a.djx_starttimetwo13,
                   a.djx_endtimetwo13,
                   c.djx_hours13
            FROM dengjx_enrollments13 e
            JOIN dengjx_teachingassignments13 a ON a.djx_assignmentid13 = e.djx_assignmentid13
            JOIN dengjx_majorcourses13 mc ON mc.djx_majorcourseid13 = a.djx_majorcourseid13
            JOIN dengjx_courses13 c ON c.djx_courseid13 = mc.djx_courseid13
            WHERE e.djx_studentid13 = ?
              AND e.djx_status13 IN ('SELECTED', 'COMPLETED')
              AND a.djx_assignmentid13 <> ?
              AND a.djx_academicyear13 = ?
              AND a.djx_semester13 = ?
            """, context.studentId(), target.getAssignmentId(), context.academicYear(), context.semester());
    for (Map<String, Object> row : rows) {
        Assignment existing = ScheduleGridBuilder.assignmentFromRow(row);
        int existingHours = numberValue(row.get("djx_hours13")).intValue();
        if (AssignmentScheduleRules.hasConflict(targetSlots, AssignmentScheduleRules.effectiveSlots(existing, existingHours))) {
            throw new BusinessException("该时间段已有课程，不能重复选课");
        }
    }
}
```

学生选课冲突判断流程可以概括为：

1. 先把当前要选的课程 `target` 按学时转换成 `targetSlots`。
2. 查询该学生当前学年、当前学期内所有有效选课记录。
3. 有效选课状态只包括 `SELECTED` 和 `COMPLETED`，不包括 `DROPPED`。
4. 查询时排除当前要选的开课安排编号，避免重复比较。
5. 遍历每一条已选课程记录，把数据库查询结果转换成 `Assignment`。
6. 读取该已选课程的学时，再转换成 `EffectiveSlot` 列表。
7. 用 `AssignmentScheduleRules.hasConflict(targetSlots, existingSlots)` 比较当前课程和已选课程。
8. 只要有任意一门已选课程发生时间重叠，就抛出“该时间段已有课程，不能重复选课”。

所以，学生选课时的实际效果是：系统会先查出该学生当前学年学期内所有有效课程，然后逐个转换为 `EffectiveSlot` 与当前要选课程比较。代码中没有先把所有已选课程统一合并成一个大 `List<EffectiveSlot>`，而是在循环中一门一门转换、一门一门判断；但逻辑效果等价于拿当前课程时间去和学生已有有效课表逐项比较。

数据库层还通过 `Dengjx_Enrollments13` 的唯一约束防止同一学生重复选择同一个教学任务：

```sql
UNIQUE (
    djx_StudentId13,
    djx_AssignmentId13
)
```

这类唯一约束解决的是“同一学生是否重复选择同一门开课安排”，而不是“不同课程之间时间是否重叠”。因此，本项目的冲突控制是数据库基础约束和后端业务判断共同完成：数据库保证字段合法和关系不重复，后端负责复杂的课程时间重叠判断。

## 视图

视图定义在 `design/sql/03_views.sql`。项目一共定义了 7 个视图，主要目的不是保存新数据，而是把验收中常见的多表联查封装成稳定查询对象，减少后端重复写复杂连接 SQL。

**1. `V_Dengjx_StudentGrades13`：学生成绩明细视图**

该视图联查成绩、选课、学生、班级、开课安排、专业课程、课程和教师，返回学号、姓名、班级、课程、教师、学年学期、课程类型、成绩、学分和录入时间。视图内部过滤 `DROPPED` 退课记录，避免退课成绩进入有效成绩列表。

代码中由 `GradeServiceImpl.studentGrades()` 查询，对应学生端接口是 `GET /api/student/grades/mine`。

**2. `V_Dengjx_TeachingEvaluations13`：教学评价明细视图**

该视图联查教学评价、选课、学生、开课安排、班级、专业课程、课程和教师，返回评分、评语、评价时间、学生、课程、班级、教师和学期信息。

当前后端 `TeachingEvaluationServiceImpl` 没有直接查询该视图，而是用等价的显式 SQL 查询评价列表。答辩时可以说明：该视图是数据库层预留的评价明细封装对象，可用于替换教师端或管理员端评价明细 SQL。

**3. `V_Dengjx_TeacherAssignments13`：教师任课视图**

该视图联查开课安排、教师、课程、班级、教室、教学楼和选课记录，返回教师任课明细，并统计有效已选人数。有效人数只统计 `SELECTED` 和 `COMPLETED` 状态。

代码中有两处使用：`StatisticsServiceImpl.teacherAssignments()` 用于管理员查看教师任课统计；`EnrollmentServiceImpl.available()` 连接该视图读取 `djx_SelectedCount13`，用于学生选课页面展示课程已选人数和容量。

**4. `V_Dengjx_ClassCourses13`：班级开课视图**

该视图联查开课安排、班级、专业、专业课程、课程、教师、教室和教学楼，返回班级维度的课程安排，包括专业、课程、教师、上课地点、学年学期、上课时间、课程类型和建议年级学期。

代码中由 `StatisticsServiceImpl.classCourses()` 查询，对应管理员接口是 `GET /api/admin/statistics/class-courses`。

**5. `V_Dengjx_CourseAverage13`：课程成绩统计视图**

该视图按每个开课安排统计平均分、最高分、最低分、通过人数和已录入成绩人数。它使用左连接成绩和选课，所以未录入成绩的开课安排也能保留在统计结果中。

代码中有两处使用：`StatisticsServiceImpl.adminCourseAverages()` 提供管理员课程成绩统计；`StatisticsServiceImpl.teacherCourseAverages()` 在该视图上按当前教师过滤，提供教师查看本人课程成绩统计。

**6. `V_Dengjx_RegionStudentCount13`：地区学生数统计视图**

该视图联查地区和学生，按生源地区统计学生数量。它使用地区表左连接学生表，所以即使某地区暂无学生，也可以显示为 0。

代码中由 `StatisticsServiceImpl.regionCounts()` 查询，对应管理员接口是 `GET /api/admin/statistics/regions`。

**7. `V_Dengjx_StudentCreditSummary13`：学生学分汇总视图**

该视图联查学生、班级、专业、选课、成绩、开课安排、专业课程和课程，返回毕业要求学分、已完成课程数、通过课程数、学生表缓存学分 `djx_TotalCredits13` 和实时计算学分 `djx_CalculatedCredits13`。

代码中有三处使用：`StatisticsServiceImpl.studentCreditSummaries()` 给管理员查看全体学生学分；`StatisticsServiceImpl.studentCredits()` 给学生查看本人学分；`AuthServiceImpl.currentProfile()` 查询当前用户资料时左连接该视图，返回学生当前学分。

从代码调用上看，项目主要通过 `JdbcTemplate.queryForList()` 直接查询视图。视图承担的是“读模型”作用：基础业务写入仍然发生在原始表，例如成绩写入 `Dengjx_Grades13`、选课写入 `Dengjx_Enrollments13`；需要展示统计或多表明细时，再通过视图统一读取。

需要注意的是，`V_Dengjx_TeachingEvaluations13` 虽然已经在 SQL 中定义，但当前 Java 代码没有直接引用。它不影响系统运行，验收时可以把它解释为教学评价明细的数据库封装对象；如果后续想进一步统一查询风格，可以把 `TeachingEvaluationServiceImpl.assignmentEvaluations()` 中的显式联查改为查询该视图并按 `assignmentId` 过滤。

## 触发器

触发器定义在 `design/sql/04_triggers.sql`，核心目标是维护 `Dengjx_Students13.djx_TotalCredits13` 这个学生已修学分缓存字段。项目没有在 Java 代码中手动调用触发器；触发器由数据库在成绩表发生插入、更新、删除时自动执行。

| 对象 | 类型 | 绑定位置 | 主要作用 |
|---|---|---|---|
| `Dengjx_RecalculateStudentCredits13(student_id BIGINT)` | 普通 PL/pgSQL 函数 | 不直接绑定表，由触发器函数调用 | 根据指定学生的有效选课和及格成绩重新计算学分，然后更新 `Dengjx_Students13.djx_TotalCredits13`。 |
| `Dengjx_GradesAfterInsertUpdateFn13()` | 触发器函数 | 被 `Dengjx_GradesAfterInsertUpdate13` 调用 | 成绩新增或修改后，根据 `NEW.djx_EnrollmentId13` 找到学生并重算学分；如果更新时成绩记录关联的选课记录发生变化，还会根据 `OLD.djx_EnrollmentId13` 找到原学生并重算原学生学分。 |
| `Dengjx_GradesAfterDeleteFn13()` | 触发器函数 | 被 `Dengjx_GradesAfterDelete13` 调用 | 成绩删除后，根据 `OLD.djx_EnrollmentId13` 找到原学生并重算学分。 |
| `Dengjx_GradesAfterInsertUpdate13` | 表触发器 | `AFTER INSERT OR UPDATE ON Dengjx_Grades13 FOR EACH ROW` | 每新增或修改一条成绩记录后，自动执行 `Dengjx_GradesAfterInsertUpdateFn13()`。 |
| `Dengjx_GradesAfterDelete13` | 表触发器 | `AFTER DELETE ON Dengjx_Grades13 FOR EACH ROW` | 每删除一条成绩记录后，自动执行 `Dengjx_GradesAfterDeleteFn13()`。 |

学分重算规则写在 `Dengjx_RecalculateStudentCredits13` 中，具体条件是：

1. 只统计当前学生的选课记录，即 `e.djx_StudentId13 = student_id`。
2. 退课记录不计入学分，即 `e.djx_Status13 <> 'DROPPED'`。
3. 只有及格成绩计入学分，即 `g.djx_Score13 >= 60`。
4. 学分值来自课程表 `Dengjx_Courses13.djx_Credit13`。
5. 如果没有任何及格课程，使用 `COALESCE(..., 0)` 把已修学分写为 0。

项目代码的触发链路是：

1. 教师端或管理员端录入成绩时，接口进入 `GradeController.teacherCreate()`、`GradeController.adminCreate()`。
2. 后端在 `GradeServiceImpl.createGrade()` 中执行 `gradeMapper.insert(grade)`，向 `Dengjx_Grades13` 插入成绩。
3. 数据库自动触发 `Dengjx_GradesAfterInsertUpdate13`，调用重算函数更新学生表缓存学分。
4. 修改成绩时，`GradeServiceImpl.teacherUpdate()` 或 `GradeServiceImpl.adminUpdate()` 执行 `gradeMapper.updateById(grade)`，数据库同样自动重算。
5. 管理员删除成绩时，`GradeServiceImpl.adminDelete()` 执行 `gradeMapper.deleteById(id)`，数据库触发 `Dengjx_GradesAfterDelete13`，把删除后的学分重新计算一遍。

这样设计的好处是：Java 业务层只负责成绩增删改，不需要在每个成绩操作后手动维护学生学分，避免遗漏。学生资料页和学分统计页读取 `djx_TotalCredits13` 时，拿到的是由数据库自动维护后的缓存值；同时 `V_Dengjx_StudentCreditSummary13` 还会返回实时计算的 `djx_CalculatedCredits13`，用于和缓存字段核对。

## 存储过程

存储过程定义在 `design/sql/05_procedures.sql`。项目定义了 2 个过程，主要用于课程设计验收中的过程化对象要求，同时也被后端统计接口实际调用。

| 存储过程 | 输入参数 | 输出参数 | 查询内容 | 项目代码如何调用 |
|---|---|---|---|---|
| `Dengjx_GetStudentYearScores13` | `student_id BIGINT`、`academic_year VARCHAR` | `result_cursor REFCURSOR` | 查询指定学生在指定学年的课程成绩，返回课程编码、课程名、学期、成绩、学分、教师工号和教师姓名。查询时过滤退课记录，并按学期、课程编码排序。 | `StatisticsServiceImpl.studentYearScores(studentId, academicYear)` 使用 `CallableStatement` 执行 `{ call Dengjx_GetStudentYearScores13(?, ?, ?) }`。管理员接口 `GET /api/admin/statistics/student-year-scores` 可传学生 ID 和学年调用；学生接口 `GET /api/student/statistics/year-scores` 会先根据当前登录用户找到本人学生 ID，再调用同一个服务方法。 |
| `Dengjx_GetCourseScoreRank13` | `assignment_id BIGINT` | `result_cursor REFCURSOR` | 查询指定开课安排内的学生成绩排名，返回学生 ID、学号、姓名、班级、成绩、排名和该开课安排平均分。排名使用 `RANK() OVER (ORDER BY g.djx_Score13 DESC)`，平均分使用窗口函数 `AVG(...) OVER ()`，查询时同样过滤退课记录。 | `StatisticsServiceImpl.courseRank(assignmentId)` 使用 `CallableStatement` 执行 `{ call Dengjx_GetCourseScoreRank13(?, ?) }`。管理员接口 `GET /api/admin/statistics/course-rank` 可直接查看；教师接口 `GET /api/teacher/statistics/course-rank` 会先校验该开课安排属于当前教师；学生接口 `GET /api/student/statistics/rank` 会先校验学生本人选过该课程。 |

后端调用存储过程的公共方法是 `StatisticsServiceImpl.callRefCursor()`。因为这两个过程通过 `REFCURSOR` 返回结果集，代码调用时会：

1. 通过 `jdbcTemplate.execute(ConnectionCallback<...>)` 取得数据库连接。
2. 先保存原来的 `autoCommit` 状态，再执行 `connection.setAutoCommit(false)`。
3. 使用 `connection.prepareCall(...)` 创建 `CallableStatement`。
4. 设置输入参数，并通过 `registerOutParameter(..., Types.REF_CURSOR)` 注册游标输出参数。
5. 执行过程后，用 `statement.getObject(outIndex)` 取得 `ResultSet`。
6. 调用 `resultSetToList()` 把结果集转成 `List<Map<String, Object>>` 返回给控制器。
7. 最后恢复原来的 `autoCommit` 状态。

这里必须关闭自动提交，是因为 openGauss/PostgreSQL 风格的游标需要在事务上下文中读取。如果直接在自动提交模式下读取 `REFCURSOR`，游标可能在语句结束后失效。

## CSV 批量导入功能

学生 CSV 批量导入是前端上传 CSV 文件，后端逐行解析并复用学生新增逻辑完成入库。

前端入口位于 `frontend/src/views/admin/StudentManagementView.vue`。学生管理页工具栏提供“CSV导入”按钮，点击后打开导入弹窗。弹窗中可以通过 `el-upload` 拖拽文件，也可以通过隐藏的原生 `input[type=file]` 选择文件。前端要求文件类型为 `.csv` 或 `text/csv`，并在界面提示 CSV 表头必须为：

```text
sno,sname,gender,age,classId,regionId,admissionDate
```

点击“开始导入”后，页面调用 `submitImport()`。如果没有选择文件，前端直接提示“请先选择CSV文件”；如果已经选择文件，则调用 `frontend/src/api/modules/admin.ts` 中的 `importStudentsFromCsv(file)`。该方法创建 `FormData`，将文件以 `file` 字段提交到后端：

```ts
const formData = new FormData()
formData.append('file', file)
return http.post('/admin/students/import', formData)
```

由于前端 HTTP 实例的 `baseURL` 为 `/api`，所以实际请求地址是：

```text
POST /api/admin/students/import
Content-Type: multipart/form-data
文件字段名：file
```

后端入口位于 `StudentController`：

```java
@PostMapping(value = "/import", consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
public ApiResponse<StudentImportResult> importCsv(@RequestParam("file") MultipartFile file) {
    return ApiResponse.ok(studentService.importCsv(file));
}
```

核心业务在 `StudentServiceImpl.importCsv()` 中。后端首先判断文件不能为空，然后使用 Apache Commons CSV 读取文件内容。读取前会按 UTF-8 解析字节，并去除 UTF-8 BOM；CSV 解析配置会启用表头、跳过表头记录、忽略空行并自动裁剪字段首尾空格。

导入前会校验表头是否包含以下字段：

1. `sno`：学号。
2. `sname`：姓名。
3. `gender`：性别。
4. `age`：年龄。
5. `classId`：班级 ID。
6. `regionId`：地区 ID。
7. `admissionDate`：入学日期。

如果缺少表头字段，后端直接返回业务异常，例如“CSV文件缺少表头字段：xxx”。如果表头完整，则逐行转换为 `StudentRequest`：

```java
return new StudentRequest(
        requiredText(record, "sno"),
        requiredText(record, "sname"),
        requiredText(record, "gender"),
        parseInteger(record, "age"),
        parseLong(record, "classId"),
        parseLong(record, "regionId"),
        parseDate(record, "admissionDate"));
```

字段校验规则包括：文本字段不能为空；`age`、`classId`、`regionId` 必须能解析为整数；`admissionDate` 必须是 `yyyy-MM-dd` 格式。转换成 `StudentRequest` 后，会复用普通学生新增逻辑 `create(request)`，因此还会继续执行学生业务校验：年龄必须在 15 到 35 之间，性别只能是 `MALE` 或 `FEMALE`，入学日期不能晚于当前日期。

学生记录创建成功后，后端会自动创建默认学生账号。账号创建逻辑在 `createDefaultStudentAccount(student)` 中：

1. 用户名为 `s` 加学号，例如学号为 `20240001`，用户名就是 `s20240001`。
2. 初始密码固定为 `123456`，但入库前会经过 `PasswordEncoder` 加密。
3. 角色为 `STUDENT`。
4. 账号绑定刚插入学生记录返回的 `studentId`。
5. `enabled` 设置为 `true`。

导入结果使用 `StudentImportResult` 返回，包含三个字段：

```java
public record StudentImportResult(
        int successCount,
        int failureCount,
        List<String> errors) {
}
```

当前实现采用“部分成功”策略。某一行导入失败不会终止整个 CSV 导入，而是记录该行错误并继续处理后续行。业务异常会直接返回具体原因，例如字段为空、年龄不合法、日期格式错误；数据库完整性异常会统一记录为“数据已存在或关联数据不存在”。错误信息会带上行号：

```java
return "第" + (record.getRecordNumber() + 1) + "行：" + message;
```

前端收到结果后会刷新学生列表，并展示成功条数、失败条数和每一行错误。如果 `failureCount > 0`，页面提示“导入完成：成功x条，失败x条”；如果没有失败，则提示“导入成功：x条”。

需要注意的是，当前 `importCsv()` 方法本身没有标注 `@Transactional`。因此它不是整批事务回滚模式，而是逐行导入、逐行记录错误的模式。如果某行学生插入成功，但后续默认账号创建失败，这一行可能被记录为失败，但学生记录已经写入数据库。答辩时可以说明这是当前实现的边界风险；如果要进一步增强一致性，可以把单行学生创建和默认账号创建放入同一个事务边界中。

**问题：转专业申请为什么不直接覆盖历史数据？**

转专业申请表会保存申请学生、原专业/班级、目标专业/班级、审核状态、审核意见和生效学期。审核通过后只更新学生当前班级，不删除历史选课、成绩和已完成课程。这样可以保留学生转专业前后的完整历史，避免成绩、选课和学分记录失真。

**问题：转专业申请和审核的完整流程是什么？**

学生端提交转专业申请时，请求地址为 `/api/student/major-transfer-applications`，后端接口位于 `MajorTransferApplicationController.submit()`，核心业务逻辑在 `MajorTransferApplicationServiceImpl.submit()` 中。

提交时系统会先检查全局转专业申请开关。如果 `Dengjx_MajorTransferSettings13.djx_Enabled13` 为 `false`，后端直接抛出“当前转专业申请未开放”。如果开关开启，系统会根据当前登录用户找到学生档案，查询学生当前班级和当前专业，并校验目标专业不能等于当前专业。同一学生如果已经存在 `PENDING` 状态的申请，也不能再次提交。

申请记录写入 `Dengjx_MajorTransferApplications13`，初始状态为 `PENDING`。记录中会保存学生编号、原专业编号、原班级编号、目标专业编号、申请理由和申请时间。

管理员审核时，请求地址为 `/api/admin/major-transfer-applications/{id}/review`，后端接口位于 `MajorTransferApplicationController.review()`，核心业务逻辑在 `MajorTransferApplicationServiceImpl.review()` 中。审核状态只允许 `APPROVED` 或 `REJECTED`。如果审核通过，管理员必须选择目标专业下的转入班级，后端会校验该班级确实属于申请中的目标专业；如果审核驳回，则不需要选择转入班级。

审核通过后，系统会立即把学生表 `Dengjx_Students13.djx_ClassId13` 更新为目标班级，同时在申请表中保存 `targetClassId`、审核意见、审核时间、生效学年和生效学期。历史选课、成绩和已完成课程不会被删除。

**问题：`fromClassId` 是什么，为什么需要保存？**

`fromClassId` 表示学生转专业前的原班级编号，对应数据库字段 `Dengjx_MajorTransferApplications13.djx_FromClassId13`。

学生提交申请时，系统会通过当前学生档案查出学生所在班级，并写入申请记录：

```java
application.setFromClassId(longValue(current.get("classid")));
```

它的作用是保留“审核通过前学生属于哪个班级”。因为审核通过后，学生表中的 `classId` 会立即更新为目标班级。如果不额外保存 `fromClassId`，系统就无法判断当前学期是否还应该按原班级处理课程、选课和课表。

因此，`fromClassId` 是实现“当前学期仍按原班级修读，下学期起按新班级生效”的关键字段。

**问题：生效学期是怎么计算出来的？**

生效学期在管理员审核通过时计算，核心代码在 `MajorTransferApplicationServiceImpl.approve()` 和 `nextTerm()` 中。

系统先通过 `AcademicTermService.current(admissionDate)` 根据学生入学日期和当前日期计算当前学年、当前年级和当前学期。学期规则如下：

1. 每年 9 月到次年 1 月算第 1 学期。
2. 每年 2 月到 8 月算第 2 学期。
3. 学年从 9 月开始，例如 2025 年 9 月到 2026 年 8 月属于 `2025-2026` 学年。

然后 `nextTerm()` 把生效时间设置为下一学期：

```java
private AcademicTermService.AcademicTerm nextTerm(AcademicTermService.AcademicTerm currentTerm) {
    if (currentTerm.semester() == 1) {
        return new AcademicTermService.AcademicTerm(
                currentTerm.grade(),
                2,
                currentTerm.academicYear(),
                currentTerm.label());
    }
    int startYear = Integer.parseInt(currentTerm.academicYear().substring(0, 4)) + 1;
    return new AcademicTermService.AcademicTerm(
            currentTerm.grade() + 1,
            1,
            startYear + "-" + (startYear + 1),
            currentTerm.label());
}
```

如果当前是第 1 学期，生效学期就是同一学年的第 2 学期。如果当前是第 2 学期，生效学期就是下一学年的第 1 学期。

例如当前日期为 2026-07-06，按规则属于 `2025-2026` 学年第 2 学期。此时审核通过转专业申请，生效学期会写为 `2026-2027` 第 1 学期。

**问题：系统如何判断当前是否已经到转专业生效学期？**

判断逻辑在 `EnrollmentServiceImpl.effectiveClassId()` 中。这个方法会被 `studentTermContext()` 调用，而 `studentTermContext()` 又会被学生选课、我的课程和我的课表等逻辑复用。

`effectiveClassId()` 会查询该学生最近一条审核通过的转专业申请：

```sql
SELECT djx_fromclassid13 AS fromclassid,
       djx_effectiveacademicyear13 AS effectiveacademicyear,
       djx_effectivesemester13 AS effectivesemester
FROM dengjx_majortransferapplications13
WHERE djx_studentid13 = ?
  AND djx_status13 = 'APPROVED'
  AND djx_fromclassid13 IS NOT NULL
  AND djx_effectiveacademicyear13 IS NOT NULL
  AND djx_effectivesemester13 IS NOT NULL
ORDER BY djx_reviewedat13 DESC
LIMIT 1
```

如果没有审核通过的转专业记录，就直接返回学生表当前班级。如果存在记录，则比较“当前学期”和“生效学期”：

```java
if (isBefore(currentTerm.academicYear(), currentTerm.semester(), effectiveAcademicYear, effectiveSemester)) {
    return numberValue(transfer.get("fromclassid")).longValue();
}
return currentClassId;
```

这里的含义是：

1. 如果当前学期早于生效学期，说明转专业还没有真正生效，返回 `fromClassId`，继续按原班级处理。
2. 如果当前学期等于或晚于生效学期，说明转专业已经生效，返回 `currentClassId`，也就是学生表里的新班级。

具体比较由 `isBefore()` 完成：

```java
private boolean isBefore(String academicYear, int semester, String effectiveAcademicYear, int effectiveSemester) {
    int currentStartYear = Integer.parseInt(academicYear.substring(0, 4));
    int effectiveStartYear = Integer.parseInt(effectiveAcademicYear.substring(0, 4));
    if (currentStartYear != effectiveStartYear) {
        return currentStartYear < effectiveStartYear;
    }
    return semester < effectiveSemester;
}
```

该方法先比较学年起始年份，例如 `2025-2026` 的起始年份是 `2025`，`2026-2027` 的起始年份是 `2026`。如果学年不同，起始年份小的更早；如果学年相同，再比较第 1 学期或第 2 学期。

举例说明：

1. 当前学期是 `2025-2026` 第 2 学期，生效学期是 `2026-2027` 第 1 学期。因为 `2025 < 2026`，所以当前还没到生效学期，系统继续使用 `fromClassId`。
2. 当前学期是 `2026-2027` 第 1 学期，生效学期也是 `2026-2027` 第 1 学期。两者相等，不属于“当前早于生效学期”，所以系统使用学生表中的当前班级，也就是新班级。

**问题：为什么学生信息会立即显示新班级，但当前学期课程仍按原班级？**

审核通过时，系统会立即更新 `Dengjx_Students13.djx_ClassId13`，所以“本人信息”“学籍状态”等直接联查学生表的页面会显示新班级和新专业。

但是选课、我的课程和我的课表不会直接使用学生表班级，而是先调用 `EnrollmentServiceImpl.studentTermContext()`，再通过 `effectiveClassId()` 判断当前学期应该使用哪个班级。生效学期到来之前返回 `fromClassId`，生效学期到来后返回学生表当前班级。

这样实现的效果是：学生档案可以先完成转专业变更，便于管理员和学生查看审核结果；教学规则仍按生效学期控制，避免当前学期课程、课表和选课被中途切换。

## 个人课表功能类

**问题：个人课表为什么不单独建一张课表表？**

个人课表不是单独维护一张课表表，而是根据开课安排、选课记录和当前登录用户身份动态生成。这样可以避免课表数据冗余，也能保证选课变化后课表同步更新。

**问题：学生个人课表是怎么实现的？**

学生端菜单中的“我的课表”对应前端路由 `/student/schedule`，页面文件为 `frontend/src/views/student/MyScheduleView.vue`。

页面加载时调用：

```ts
studentGet<ScheduleResponse>('/schedule')
```

`studentGet` 会统一拼接 `/student` 前缀，所以实际请求地址为 `/api/student/schedule`。后端接口位于 `EnrollmentController`：

```java
@GetMapping("/api/student/schedule")
public ApiResponse<Map<String, Object>> schedule(@AuthenticationPrincipal AuthenticatedUser user) {
    return ApiResponse.ok(enrollmentService.schedule(user.userId()));
}
```

后端核心逻辑在 `EnrollmentServiceImpl.schedule()` 中：先根据当前登录用户 `userId` 查询绑定的学生编号，再复用 `mine(userId)` 查询当前学生本学年、本学期的有效选课记录。有效选课状态只包含 `SELECTED` 和 `COMPLETED`，退课记录不进入课表。查询结果包含课程名称、教师、教室、学年学期、课程类型、上课星期和上课时间等信息，最后调用 `ScheduleGridBuilder.build(rows)` 把课程列表转换为课表结构。

`mine(userId)` 在查询前还会根据学生入学时间计算当前年级、当前学期和当前学年，并自动补齐当前学期应修的必修课选课记录。这样学生即使没有手动选择必修课，也能在“我的课程”和“我的课表”中看到必修课程。

**问题：教师个人课表和学生课表有什么区别？**

教师端“我的课表”对应前端路由 `/teacher/schedule`，页面文件为 `frontend/src/views/teacher/TeacherScheduleView.vue`。前端展示方式与学生课表基本一致，也是按前半学期、后半学期、星期和节次渲染。

不同点在于数据来源。教师课表请求 `/api/teacher/schedule`，后端接口位于 `StatisticsController`：

```java
@GetMapping("/api/teacher/schedule")
public ApiResponse<Map<String, Object>> teacherSchedule(@AuthenticationPrincipal AuthenticatedUser user) {
    return ApiResponse.ok(statisticsService.teacherSchedule(user.userId()));
}
```

`StatisticsServiceImpl.teacherSchedule()` 会先根据当前登录账号找到绑定的教师编号，然后查询该教师负责的所有开课安排。查询结果同样包含课程、班级、教室、学年学期和上课时间，最后调用 `ScheduleGridBuilder.build()` 生成和学生课表相同的数据结构。

**问题：课表如何区分前半学期和后半学期？**

课表拆分规则由 `AssignmentScheduleRules.effectiveSlots()` 统一处理。系统会根据课程学时把开课安排转换成真正有效的上课时间段：

1. `32` 学时课程只使用第一个上课时间，并且只在前半学期有效。
2. `48` 学时课程使用两个上课时间，第一个时间全学期有效，第二个时间只在前半学期有效。
3. `64` 学时课程使用两个上课时间，并且两个时间都全学期有效。

`ScheduleGridBuilder` 根据这些有效时间段，把课程分别放入 `firstHalf` 和 `secondHalf`。前端不再计算复杂规则，只负责根据后端返回结果展示课表。

**问题：课表接口返回什么结构，前端如何展示？**

`ScheduleGridBuilder.build()` 会返回一个 `Map.of("firstHalf", firstHalf, "secondHalf", secondHalf)`。其中 `firstHalf` 和 `secondHalf` 都是按星期分组的 Map，后端初始化时会固定生成周一到周五 5 个键，所以即使某天没有课程，也会返回空数组。
